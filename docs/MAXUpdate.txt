1. Introduction 

 

1.1 MAXUpdate Overview 

MAXUpdate is a package that includes a programmer’s reference and sample code for the MAX 
Manufacturing Software package. This Application Programmer’s Interface (API) package should 
allow an experienced programmer to build custom interfaces that interact with MAX function code 
without having to worry about how to update the MAX database. 

MAXUpdate was developed to extend the foundation built by the MAX for Windows modules. At 
present, MAXUpdate supports Sales Order, Purchase Order and Shop Order Entry as well as 
Shipping, Purchasing, Inventory and Shop Floor Control transaction functions. 

MAXUpdate provides the experienced programmer with tools and an example method with which 
to program applications that “talk to” the MAX database. Generally professional work experience in 
.NET software development is recommended before considering working at this level. Without 
prior experience, the learning curve is steep. It is highly recommended to have experience in both 
programming and database management in order to use MAXUpdate. 

 

1.2 Scope 

The integration components included within MAXUpdate are not a sufficient tool set to integrate a 
CRM application with MAX. MAXUpdate describes how to call a specific set of order and 
transaction functions that obey MAX business rules to update the MAX database. 

MAXUpdate is used ideally in situations where the programmer needs the database functionality of 
MAX, but would like to craft custom user interfaces using the development platform of their 
choosing. It would be suitable under platforms such as .NET, Visual Basic, C++, etc. 

Since MAXUpdate is internally maintained by ECI, the application programmer is relieved from the 
chore of updating the Data Base routines whenever there is a logic or DB field change. This allows 
the programmer to focus on the user interface (screen) aspect of the application. On the other 
hand, the programmer using MAXUpdate does not have access to the details of the code 
contained within the DLLs. This means that custom logic changes for a particular application 
cannot be accommodated. 

 

1.3 Expectations 

When making function calls to MAXUpdate all fields need to be initialized and passed valid data. 
This is a tool for developers familiar with how MAX works and what data it requires. The 
MAXUpdate calls do not do lookups for default values for any fields (unless specifically mentioned 
in the descriptions) and do not fill in any default values into the fields. Some fields do have 
validation to make sure completely invalid data is not passed however many do not. The best way 
to validate what data is required by MAX would be to create a record using the MAX UI and then 
looking in the database to verify what fields were filled in, all fields that are not a blank string must 
be passed in to the MAXUpdate function call. It is possible using MAXUpdate to populate data into 
the database which will cause certain MAX functionality to break, it is the responsibility of the 
developer to ensure that their application is correctly populating all fields. 

The MAX Technical Support agreement does not cover corrupt data caused by passing invalid 
data into a MAXUpdate function. Before calling support on an issue with any orders created or 
modified using MAXUpdate you should always verify the problem can be duplicated on a record 
that has not been affected by your application which uses MAXUpdate. If the problem cannot be 
duplicated please verify what values are populated in the database when using the MAX UI versus 
what is populated using the custom application, and make sure your application is populating all 
values are being passed correctly. Only contact support after verifying that complete and correct 
data is being passed to the MAXUpdate function and it is still populating the database differently 


than the MAX UI does. If the problem cannot be duplicated within the MAX UI, and you have 
validated that the values populated in the database from the code match those populated by 
entering the data through the MAX UI, then the problem exists with the custom code and is outside 
the scope of the support coverage. For assistance troubleshooting your custom code, please 
contact the services team to assist you with the code or setup of MAXUpdate. 

 

 

 


1.4 DLL Dependency Diagram 

The following diagram shows the interdependencies of the Dynamic Link Libraries (DLLs) used by 
the MAX for Windows system. 

Using DLLs has at least three key advantages over using a static library: 1) The DLL can be used 
by multiple "client" applications, saving memory and gaining speed, 2) All common code is 
centralized in the DLLs, which greatly reduces logic errors, 3) Should the DLL need to be revised, 
only it has to be replaced, not the entire application. 

MAX modules make calls to the libraries from the top down; i.e. a module can call any module 
which is below it on the chart, but never vice-versa. 

The MAXUpdateXML wrapper is an optional layer that can be used to simplify the data 
manipulation required for MAXUpdate by allowing the user to pass XML data rather than formatted 
data structures. Therefore, in general, when new functionality is added to MAX any applications 
calling the MAXUpdateXML wrapper will continue to function without any changes, where an 
application not using the wrapper will at a minimum need to be recompiled with the new data 
structure changes and re-distributed. 

NOTE: When programming in a .NET language it is required to use the MAXUpdateXML wrapper, 
.NET does not allow defining a data structure that will work with the standard MAXUpdate DLL 
calls.

2. MAX DLL Descriptions 

MAXUPDATEXML - The MAXUpdateXML DLL is an alternate interface to the DLLs listed below. 
Instead of formatted structures the functions calls in this DLL take XML strings. When 
programming in a .NET language the other DLLs will not function properly and this is the only 
interface available for MAXUpdate. 

MAXORDR2 - The MAXOrdr2 DLL is used to update orders in the MAX system. This includes 
adding, changing and deleting line items where applicable (e.g. Sales Orders). Currently, the DLL 
includes functionality for Sales Orders, Forecast Orders and Purchase Orders 

MAXTRAN2 - The MAX Transaction DLL handles inventory and shop floor control transactions. 
Examples include Cycle Count, Stock transfer, Receipt to Stock, Completions, Time Ticket 
operations, etc. This library is unique in that all transactions are accessed through a general-
purpose function call (See the Function Description for additional information). 

MAXEXEC2 - This is the low-level interface to the MAX database. It contains many support and 
utility routines that are used by the higher-level calls. Currently, these base routines are not 
documented or supported in the MAX Update package.

5. XML Table Structure 

The XML structure uses the same field names and table names as currently defined in the SQL 
Database. The same XML structure is used by the ETL module. 

The XML should be defined using version 1.0 specifications and utf-8 encoding. 

The root element is always "eMAXExact", followed by the table name tag, the record name tag and 
then the actual fields. The basic outline should look like : 

<?xml version="1.0" encoding="utf-8" ?> 

<eMAXExact> 

 <TableName_Table> 

 <TableName> 

… 

 </TableName> 

 </TableName_Table> 

</eMAXExact> 

 

The exact formatting of the XML Structure is fairly flexible as defined in the XML Spec, all the 
examples will be structured as above for readability, however passing the following XML will also 
work equally well: 

<?xml version="1.0" encoding="utf-8" ?> <eMAXExact> <TableName_Table> <TableName> … 
</TableName> </TableName_Table> </eMAXExact> 

 

5.1 Formatting of XML Data 

There are basically 3 data types that will be passed in the XML strings, dates and numbers. Be 
sure to check the table definitions either in SQL or the Database reference guide for the data type 
of a given field. 

5.1.1 Passing Strings in XML 

In MAX most fields do not allow null values so it is highly recommended to not pass an XML value 
that indicates null when calling the MAXUpdate functions. To indicate a blank string for the field 
DISPOSITION_39 either do not include the field at all in the XML or use the following are format: 

<DISPOSITION_39></DISPOSITION_39> 

Do not use the shortcut definition <DISPOSITION_39/> as this is interpreted as a null value and in 
many fields will cause an error to be returned. 

5.1.2 Passing Dates in XML 

In most fields requiring a date the format used should be yyyy-MM-dd, so February 3, 2011 would 
be 2011-02-03. The exception to this is the TNXDTE_39 field for processing transactions, this date 
should be passed in the format MMddyy, so the above example would be 020311. When a null 
date is required either do not pass the field in the XML or use 0001-01-01 as the date. 

5.1.3 Passing Numbers in XML 

For numeric fields in MAX simply pass the string equivalent of the number (in base 10 only, MAX 
does not recognize hex or octal numbering).

6. Transaction Rules 

6.1 General 

Both Type and SubType are required for all transactions. If SubType is shown as blank, that 
means a blank space should be passed into the SubType ‘ ‘, not an empty string ‘’. 

The Transaction Date and Transaction Time are optional fields, the current date/time of the system 
will be used if they are not passed in the XML. Usually it is best to leave these out of the XML 
since even if passed they will only be used if the system switch “Trans Date Time” is set to Y. 

In all cases the Lot/Serial information must be passed using quantities in the BOM UOM for the 
part. Some transactions (see notes in Sections 6.7.x to identify transactions that require a UOM 
other than the BOM UOM) have a transaction quantity in the Purchasing UOM, even in these 
cases the Lot/Serial information must be in BOM UOM as this is the UOM used by MAX for all 
stock information. 

6.2 Lot/Serial Tracking (XML) 

Lot/Serial information must be passed in transactions for all parts under Lot/Serial control. The 
XML examples in the previous section show the different ways this information can be passed. 
For a part under Serial control the Assignment Code field (ASSCODE_39) must be I (Individual), 
or R (Range). If R is selected it indicates the serial numbers will be assigned sequentially starting 
with the serial number passed in SERIAL_39. The Serial number passed must follow the standard 
MAX rules for auto assigned serial numbers. If I is selected it indicates the individual serial 
numbers will be contained in the <SerialNumbers> section of the XML. 

For Cycle Count and zero quantity Adjustment transactions the Lot/Serial information passed must 
contain the total Lot/Serial information for the stockroom being transacted. This is not a delta 
quantity, the entire on-hand quantity for all lots/serials after the transaction completes must be 
passed in the XML. In these cases the R Assignment code is not recommended as it is likely to 
cause unexpected results. 

6.3 Lot/Serial Tracking (Data Structures) 

Either the lot number and/or the serial number field must be filled in. If serial tracking is used the 
Assignment Code for Serial numbers must be I (Individual), or R (Range) in the serial number field. 
If R is selected the serial number for the first serial number should be entered in the serial number 
field - the remaining serial numbers will be automatically generated using the transaction quantity 
(only serial numbers ending in a numeric value can be used for the range option). The I option 
requires that an array of serial numbers be passed. The method for this is as follows: 

6.3.1 Serial Tracking (Individual Option) 

Three steps are required in addition to making the transaction call. The first step is to call 
CreateSerialArray with the number of serial records to be created. This allocates enough memory 
to hold the serial numbers. The next step is to call SetSerialNumber for each serial number to be 
created. Then call ProcessTrans to do the desired transaction. The last step is to call 
DeleteSerialArray to free up the memory allocated by CreateSerialArray. Be sure to set the 
“Assignment code for serial #’s” field to ‘I’ for individual. 

6.3.2 Lot and Serial Tracking (Individual Option) 

These are handles as above except using the functions CreateLotAndSerialArray, 
SetLotAndSerialNumber and DeleteLotAndSerialArray. 


6.3.3 Cycle Count (Subtype L) 

This applies to cycle counts with lots only. Cycle counts with serial numbers only use the 
procedure above. Cycle counts with Lot and Serials can not be done using this procedure unless 
the “R” option is used for serial numbers. 

As for individual serial numbers, three steps are required in addition to making the transaction call. 
The first step is to call CreateLotQtyArray with the number of lot records to be created. This 
allocates enough memory to hold the lot numbers. The next step is to call SetLotQty for each lot 
number to be created. Then call ProcessTrans to do the cycle count transaction. The last step is 
to call DeleteLotQtyArray to free up the memory allocated by CreateLotQtyArray.

7. Usage 

To use the MAX Update package for development, the following steps should be taken: 

1. We recommend VB.NET or C#.NET as the development language. 
2. Make sure the MAX client is installed on the development machine and add the 
RMClient\EFW folder into your system PATH statement. 
3. In the .NET environment add ExactRMEnc.DLL, ERMRemCl.DLL and 
MaxUpdateXML.DLL as references to your project. (ExactRMEnc and ERMRemCl are not 
directly called, but adding them as references will cause them to automatically copy to the 
EXE directory allowing Debugging to work easily). 
4. Copy the .dll, and the appropriate include files to your development directory. For the 
.NET environment the above step handles this automatically. 


In the VB6 environment the ExactRMEnc.DLL and ERMRemCl.DLL must be copied into 
the C:\Program Files\Microsoft Visual Studio\VB98 directory to allow debugging from 
within the development environment. 

For other environments the DLLs must be in the directory of the running EXE in order for 
MAXUpdate calls to work correctly. 

5. If you are developing in C, you can use the header files directly without modification. 
Visual Basic developers will be able to use the .str files directly, but will need to write 
DECLARE statements for the functions they will be calling. C++ programmers must use 
the "wrapper" functions described above in the Function Description. 


If you are developing from another environment, you will need to declare the above 
functions according to the rules of that environment. The functions are for (32-bit) C-style 
calls with left-to-right parameter passing. The called function takes care of stack clean up. 

 

Note to developers: Repeated use of the Wrapper is best done by minimizing the number of 
object creation instances. After creating the object, any number of function calls can be made. 
Conversely, repeatedly creating and destroying the Wrapper object between calls can result in the 
following error: Error 91: "Object variable or With block variable not set". 

In the ASP environment, developers can choose between application, session or global object 
based on their application and single or multi-user environment. Generally, we recommend a 
session object to avoid transaction conflicts and deadlocks. 


8. VB6 Programming with MAXUpdate. 

8.1 Introduction 

Microsoft Visual Basic, Access, Excel and Word are examples of applications that support the 
Visual Basic for applications programming language. They are all applications that contain the 
necessary programming tools to call Dynamic Link Libraries (DLLs). 

The primary purpose of MAXUpdate in a Visual Basic application is to simplify the programming 
procedure necessary to update your MAX database. Even with a minimal understanding of the file 
relationships and data structures in MAX, you can successfully perform complex tasks. 

MAXUpdate also provides the necessary control to ensure data integrity, and it provides the Visual 
Basic developer with the assurance that his or her application follows all data relationship rules 
required in MAX . 

The first step in developing your application is to determine the type of MAX data updates you will 
perform. With this in mind, you can determine if you need only one of the libraries or both. When 
developing your application, only include the library that you will use. For example, including 
MAXORDR2.DLL in your application when you plan on performing only inventory transactions is 
unnecessary. In this case only include the MAXTRAN2.DLL etc. 

MAXUpdate is primarily a tool used to create user interfaces that add, update and delete data in 
the MAX database. You will however also probably have the need to retrieve and display data from 
the MAX database. Only limited data retrieval capabilities exist in the MAXUpdate Dynamic Link 
Libraries. There are several ways to retrieve data outside MAXUpdate, and you should choose the 
method that you feel the most comfortable with and that provides adequate performance. Also, 
note that any method you use to retrieve data from MAX should not create table or record locks on 
the database, unexpected behavior such as lockups and timeouts of the MAX Client on other 
workstations can be caused by a custom application locking records with an improperly designed 
database query.

We expose the Extract and Load functionality to MAX Programmers and Power Users in

The MaxDataSchema.xsd file describes the XML Schema for the data extracted and loaded by 
MAX ETL.

11.1 Setting up the ASP.NET environment 

The biggest difference between using MAXUpdate in an ASP environment and as a standalone 
EXE is the additional security setup required to allow it to run. First, log in to the console of the 
server as the user which the ASP application is running as and verify that the user has sufficient 
rights to launch and use the MAX client. For the ASP.NET environment you will need to set up a 
user whit the rights listed below, the Application Pool for your website will need to be set up to run 
as this user. 

11.1.1 File Permissions: 

The user needs read rights to the following folders (SysWOW64 is for 64-bit OSs only): 

• \\MAXServer\Exact_Shared\Exact\RMServer\LIC 
• C:\Windows\SysWOW64\config\systemprofile\AppData 
• C:\Windows\System32\config\systemprofile\AppData 


The user also needs read/write rights to the following folders: 

• \\MAXServer\Exact_Shared\Exact\RMServer\LOG 
• C:\Exact\RMClient\EFW 


11.1.2 SQL Server Permissions: 

The user needs the VIEW SERVER STATE right on the SQL Server as well as db_datareader and 
db_datawriter roles on the ExactMAX and company databases. 

11.1.3 DLLs 

The ERMRemCl.DLL and ExactRMEnc.DLL need to be copied to the appropriate location where 
the “executable” will be running from. In the ASP.NET environment this is the 
C:\Windows\System32\inetsrv folder, if your server is running a 64-bit OS they need to be copied 
to C:\Windows\SysWOW64\inetsrv instead. 

11.1.4 IIS Application Pool 

The Application Pool for the webpage that will be using MAXUpdate needs to run as a specific 
user as mentioned above, also it needs to be properly configured to use .NET Framework Version 
2.0 as well as being set to Enable 32-Bit Application (if you are running a 64-bit OS).

Note: For 32-bit Applications: 


The functions that require a pointer to an input record (such as AddSalesOder or

alignment.

would call the CreateLotAndSerialArray function, then call SetLotQty, then SetSerialNumber fifty 
times, to set all 50 serial numbers, then the ProcessTrans Function and then 
DeleteLotAndSerialArray. After that is complete the process would be repeated for the lot with 
quantity 150. When using ProcessTransXML this is all taken care of automatically in the 
background by the logic in the XML function.

15. Glossary 

DLL 

Dynamic Link Library. A Windows library which resides in memory and can be called from multiple 
client applications. Loading and Unloading is handled automatically by Windows. 

DLLEXPORT 

Defined as FAR PASCAL. This means that the DLL function must be called far (32-bit address 
call) and the parameters are passed in the Pascal language fashion (parameters pushed on the 
stack from left to right, called function does stack cleanup). 

DOUBLE 

A double is an 8-byte floating point number. 

LPCSTR 

Defined as a const char far *. A 32-bit pointer to an ASCIIZ (null-terminated) string which can not 
be changed. 

LPINT 

Defined as an int far*. A 32-bit pointer to an integer (2 Byte). 

LPLONG 

Defined as a long far*. A 32-bit pointer to a long (4 Byte) integer. 

LPSTR 

Defined as a char far *. A 32-bit pointer to an ASCIIZ (null-terminated) string. 

MAXEXEC2 

Low-level MAX Database DLL. Support and utility routines for 32 bit applications. 

MAXORDR2 

32 bit DLL formerly known as MAXOrder.dll. Code for dealing with the various types of MAX 
Orders. 

MAXTRAN2 

MAX Transaction DLL. Code for processing MAX Inventory and Shop Floor Control Transactions. 

OBJHANDLE 

Defined as void far *. A 32-bit (segment:offset) pointer. This is the address of the Object Handle 
which must be used in calls to the DLLs.

16. Frequently Asked Questions 

 

1. When I create a Sales Order using AddSalesOrderXML every field gets populated except 
ORDDTE_27. 
• This field must be set in the XML which is passed into the AddSalesOrderXML function. 


 

2. If I leave the ORDNUM_27 field blank, how does it determine what order number to use? 
Does it update the Configuration Data with the next available order number? 
• If a blank order number is passed in the structure, the AddSalesOrderXML routine will use 
the next number from the Configuration Data. It also increments the next sales order 
number in the Configuration Data. If an order number is passed that already exists, the 
AddSalesOrderXML routine will continually increment the sales order number until it finds 
the next available sales order number to use. 
3. If I let AddSalesOrderXML assign the order number, how do I retrieve that number? 
• The return value of the AddSalesOrderXML function is a string which is the order number 
which was created. A blank string is returned in the case of an error. 
4. If I let MAXUpdate generate the next order number is there a way to turn off or trap the 
message box that tells me the next number is being used? 
• Yes, the AddSOXML function has a parameter to suppress all messages, it behaves just 
as AddSalesOrderXML in all other ways. 
5. Where do you recommend installing the DLL's? 
• We recommend completing an MAX client install on the workstation, verifying that this 
install works by logging into System Manager, and then using the DLLs in the EFW folder. 
6. Does MAXUpdate allow processing Rework and/or Split functions? 
• Split orders or rework orders have no API support in Windows. 
7. What development tools can be used with MAXUpdate? 
• Any development tool that allows the calling of functions within a DLL can be used with 
MAXUpdate. The most common tools are the .NET languages, Visual Basic and C++. 
8. Do serial numbers need padding? 
• Serial numbers do require right space padding also it is recommended to use right space 
padding for all strings when using MAXUpdate. Thus the serial number 1011 needs should 
be (using C# syntax) "1011".PadRight(30).



17. Common Problems 

17.1 Application will not compile 

• All MAX *.DLL components used in MAXUpdate must be located in the 
C:\Exact\RMClient\EFW folder. 
• The development machine must have a path to the EFW folder. 


17.2 Application does not run 

• All MaxDlls must be considered as a set, MaxTran2.dll, MaxOrdr2.dll, MaxExec.dll, 
MaxSec.dll. They should be kept together 
• The ERMRemCl.DLL and ExactRMEnc.DLL must be in the same directory as the 
application EXE file. 
• Generally, your VB application must be in the EFW folder in order for your application to 
find MaxTrn2 or MaxOrdr2. Otherwise you will need a path statement for your application 
to find these DLLs. 
• Failure to initialize MAXTransItem structure. Fields in this structure must be initialized 
before calling ProcessTrans. Failure to do so may result in un-initialized data being stored 
in reference fields. The resulting data will not be visible in some MAX screens, but will 
look normal when data is viewed directly in database. See ClearMAXTransItem in the 
Visual Basic sample for an example of how this works. This condition is due to the fact that 
Visual Basic does not initialize the elements of a structure. 


17.3 Lot / Serial Control Problems 

We have developed a sample application in which we use the functions from the 
Maxtran2.dll for a Purchase Order Receipt. See this application for examples. 

17.4 Error Messages in Application 

If your application runs but crashes in the same location, add the error handling routine 
from the sample code. Match the error code with the following table. For a large robust 
application, add the sample error handling code into your application and write a function 
to display the associated error test messages in the following table. Smaller applications 
don’t need this additional work. 

17.5 MAXUpdate Version Control 

For consistent operation with MAX ERP, new MAX Update DLLs’ are provided with MAX 
Service packs. Your custom application programs in production using one version of MAX 
Update DLLs’ are generally compatible to new versions of MAXUpdate DLLs’. However it 
is good practice to test your specific custom application against a new release of 
MAXUpdate DLLs’. 

17.6 Feedback 

If you feel we have missed some important information please contact Exact Tech Support 
by entering a MAX Enhancement Request on the Exact Customer Portal.
